# Чек-лист готовності проєкту Assistant T800

Документ звіряє поточну реалізацію проєкту з вимогами фінального командного завдання GoIT. Позначення:

- ✅ реалізовано в поточній версії;
- 🟡 частково реалізовано або підготовлено архітектурно;
- ❌ ще потрібно реалізувати.

---

## 1. Основна частина проєкту

### 1.1. Зберігання та управління контактами

| Статус | Вимога | Поточний стан |
| --- | --- | --- |
| ✅ | Додавання контактів з іменем, адресою, телефонами, e-mail та днем народження | Команда `add <name> [phone] [email] [address] [birthday]` створює контакт. Додаткові поля можна передавати у будь-якому порядку. |
| ✅ | Пошук контактів за різними критеріями | Реалізовано загальний пошук `search <query>` та окремі команди `search-name`, `search-phone`, `search-email`, `search-note`, `search-tag`. |
| ✅ | Редагування контактів | Реалізовано `set-address`, `set-birthday`, `add-phone`, `add-email`. Для телефонів та e-mail підтримується передача кількох значень через `;`. |
| ✅ | Видалення контактів | Реалізовано `remove <name>` із обов’язковим підтвердженням. |
| ✅ | Видалення окремих полів контакту | Реалізовано `remove-address`, `remove-birthday`, `remove-phone`, `remove-email`. Для `remove-phone` та `remove-email` значення можна не вказувати: якщо значення одне — воно видаляється, якщо кілька — система запитує підтвердження видалення всіх. |
| ✅ | Виведення всіх контактів | Реалізовано команду `contacts`. |
| ✅ | Перегляд одного контакту | Реалізовано команду `get <name>`. |
| ✅ | Виведення контактів, у яких день народження через задану кількість днів | Реалізовано `birthdays [days]`. Якщо кількість днів не вказана, використовується 7 днів. Дата привітання переноситься з вихідних на понеділок. |
| ✅ | Валідація номера телефону під час додавання або редагування | Валідація винесена в доменне поле `Phone`. |
| ✅ | Валідація e-mail під час додавання або редагування | Валідація винесена в доменне поле `Email`. |
| ✅ | Валідація дня народження | Валідація винесена в доменне поле `Birthday`; підтримуються формати `DD.MM.YYYY`, `DD-MM-YYYY`, `DD/MM/YYYY`. |

### 1.2. Зберігання та управління нотатками

| Статус | Вимога | Поточний стан |
| --- | --- | --- |
| 🟡 | Можливість додавання текстових нотаток | У доменній моделі `Contact` вже є поле `note`, але окрема CLI-команда для редагування нотатки ще не реалізована. Запланована команда: `edit-note <name>`. |
| 🟡 | Пошук за нотатками | Реалізовано `search-note <query>`, який шукає по полю `note` контакту. Повноцінний сценарій редагування нотатки ще потрібно додати. |
| ❌ | Редагування нотаток | Заплановано через інтерактивну команду `edit-note <name>`. |
| ❌ | Видалення нотаток | Заплановано `remove-note <name>` із підтвердженням; внутрішньо нотатка має очищатися до службового порожнього значення. |

### 1.3. Зберігання даних

| Статус | Вимога | Поточний стан |
| --- | --- | --- |
| ✅ | Дані повинні зберігатися на жорсткому диску | Реалізовано pickle-сховище `AssistantStorage`. Дані зберігаються в `.data/address_book.pkl`. |
| ✅ | Помічник може бути перезапущений без втрати даних | Дані завантажуються при старті CLI/TUI та зберігаються при завершенні роботи. |
| ✅ | Історія команд | Для CLI використовується `.data/cli_commands_history`. |

### 1.4. Інтерфейс командного рядка

| Статус | Вимога | Поточний стан |
| --- | --- | --- |
| ✅ | Основний інтерфейс у вигляді командного рядка | Реалізовано класичний CLI-режим за замовчуванням. |
| ✅ | Цикл обробки команд до виходу | Реалізовано `CliRunner`, який обробляє команди до `exit`, `quit` або `q`. |
| ✅ | Команда допомоги | Реалізовано `help`. |
| ✅ | Коректна обробка некоректного вводу | Некоректний синтаксис, невідомі команди та помилки валідації повертаються як структуровані результати. |
| ✅ | Підтвердження небезпечних дій | Усі destructive-команди мають підтвердження. |
| ✅ | Підказка команд, історія та автодоповнення | Підключено `prompt_toolkit` через `InputFactory`. |
| ✅ | Підказки для невідомих команд | Підключено `suggestions` із fuzzy-пошуком та AI fallback. |

## 2. Додаткова частина проєкту

### 2.1. Теги

| Статус | Вимога | Поточний стан |
| --- | --- | --- |
| 🟡 | Додавання тегів до нотаток | У доменній моделі `Contact` вже є `tags`, але CLI-команди `add-tag` / `remove-tag` ще не реалізовані. У поточній спрощеній моделі теги планується прив’язувати до контакту. |
| ✅ | Пошук за тегами | Реалізовано `search-tag <query>`, який шукає по тегах контакту. |
| ❌ | Сортування нотаток за тегами | Не реалізовано. Можна додати після реалізації команд керування тегами. |

### 2.2. Інтелектуальний аналіз

| Статус | Вимога | Поточний стан |
| --- | --- | --- |
| ✅ | Помічник пропонує найближчу команду за введеним текстом | Реалізовано `suggestions`: fuzzy-підказки та AI-розпізнавання природного вводу. Приклад: `видали Аліса` може бути перетворено на `remove Аліса`. |
| ✅ | Підтвердження запропонованої команди | Можна налаштовувати через `force_suggestion_confirm`. У поточній CLI-збірці дозволено автоматичне виконання запропонованої команди з інформаційним повідомленням. |
| ✅ | Підтримка AI-моделі Gemini | Використовується `pydantic-ai` та `google-genai`; модель задається через `.env`, наприклад `GOOGLE_API_MODEL=google:gemini-3.1-flash-lite`. |

## 3. Додатковий функціонал понад чек-лист

| Функціонал | Статус | Опис |
| --- | --- | --- |
| Локалізація | ✅ | Повідомлення винесені в `localization/locales/*.ini`. Підтримано українську та англійську мови. |
| Мультирежимний запуск | ✅ | CLI запускається за замовчуванням, TUI — через `tui`, `--tui` або `--enable-ai`. |
| TUI-режим | ✅ | Є окремий Textual-інтерфейс із AI-чатом. Це додатковий інтерфейс, CLI залишається основним. |
| Єдиний command layer | ✅ | CLI і майбутні інтерфейси можуть працювати через спільний application dispatcher. |
| Довгі aliases команд | ✅ | Dispatcher підтримує пошук найдовшого збігу команди або alias. Наприклад, `знайди телефон Аліса` коректно потрапляє в `search-phone`. |
| Безпечне видалення | ✅ | Видалення контакту, полів, телефонів та e-mail потребує підтвердження. |
| Автоматичне збереження | ✅ | Збереження відбувається через context manager `AssistantStorage`. |
| Зручний CLI-ввід | ✅ | Підтримано історію команд, автодоповнення та fallback до стандартного `input()`. |

---

## 4. Що ще потрібно зробити

1. Додати інтерактивне редагування нотатки: `edit-note <name>`.
2. Додати очищення нотатки: `remove-note <name>`.
3. Додати керування тегами: `add-tag <name> <tag>` та `remove-tag <name> <tag>`.
4. За потреби додати сортування або групування за тегами.
5. Додати тести для основних сценаріїв CLI та application layer.
6. Перевірити фінальний README після завершення команд нотаток і тегів.
