# Персональний Assistant T800

Архітектурне бачення фінального командного проєкту.

> **Важлива примітка.** Цей документ **не** описує «enterprise-систему», яку потрібно реалізувати повністю. Його задача — зафіксувати напрямок архітектури, розділити відповідальності між частинами системи, зробити код зрозумілішим для командної роботи, підготувати проєкт до презентації та закласти можливість подальшого розширення. Ми проєктуємо не «корпоративну CRM», а гармонійну систему навчального рівня курсу.

Деякі компоненти системи можуть бути реалізовані частково, у спрощеному вигляді, як proof-of-concept або як підготовлена структура без повної реалізації.

**Головна мета проєкту:** продемонструвати інженерне та архітектурне мислення, показати правильне розділення відповідальностей, якісний код та командну організацію.

---

## Загальна концепція

Мета проєкту — створити модульний та розширюваний **Персональний помічник** на Python.

### Основні можливості

- **Контакти** та **Нотатки**
- **Теги** та функціонал **Пошуку**
- **Збереження даних** (persistence)
- **Розширюваний інтерфейс**

### Розділення відповідальностей

Кожна частина системи відповідає виключно за свою задачу:

| Шар | Призначення |
| --- | --- |
| `domain` | доменні моделі та поля |
| `repositories` | робота з колекціями даних |
| `services` | бізнес-логіка |
| `interfaces` | взаємодія з користувачем |
| `storage` | збереження даних на диск |

**Що це дає команді?** Простіше працювати над кодом разом, легше підтримувати проєкт, можливість розширювати функціонал без ризику «поламати все», а також легка заміна окремих компонентів.

## Обсяг проєкту (Scope)

### Must Have (обов’язково)

- Контакти (`contacts`) та нотатки (`notes`)
- Теги (`tags`) та пошук (`search`)
- Збереження даних (`persistence`) та валідація (`validation`)

### Nice To Have (бажано)

- AI-ready сервіс тегів
- Прототип TUI інтерфейсу (Textual prototype)
- Прототип REST API
- Підготовка до семантичного пошуку

### Чого свідомо уникаємо

- Надмірного ускладнення архітектури (*overengineering*)
- Непотрібних фреймворків
- Передчасної оптимізації
- Складності рівня великих корпоративних систем (*enterprise complexity*)

## Детальний функціонал компонентів

### 1. Контакти

- Створення, редагування та видалення контактів
- Пошук по контактах
- Відстеження днів народження

### 2. Нотатки та теги

- Нотатки мають прив’язку до контактів

### 3. Сховище (Persistence)

- Автоматичне збереження та завантаження даних
- Робота у локальному сховищі користувача

## Технічна структура проєкту

Проєкт побудований як встановлюваний Python-пакет із чітким розподілом по шарах та окремими точками входу для різних інтерфейсів користувача:

```text
project-root/
├── pyproject.toml
├── README.md
├── requirements.txt
├── src/
│   └── personal_assistant/
│       ├── __init__.py
│       ├── bootstrap.py
│       ├── domain/          — Ядро системи (доменні моделі та поля)
│       │   ├── contacts.py  (Name, Phone[], Email[], Address, Birthday)
│       │   ├── notes.py     (id, text, tags, contact_key)
│       │   └── fields.py
│       ├── repositories/    — Робота з колекціями даних
│       │   ├── contacts.py  (set, add, get, remove, find, all, exists, ordered)
│       │   └── notes.py     (set, add, get, remove, find, all, find_by_tag, ordered)
│       ├── services/        — Бізнес-логіка («мозок» системи)
│       │   ├── contacts.py  (add_contact, update_contact, delete_contact, search, birthdays)
│       │   ├── notes.py     (add_note, update_note, delete_note, search_note, find_by_tag)
│       │   └── tags.py      (suggest_tags)
│       ├── application/     — Єдина система команд
│       │   ├── commands.py  (AddContactCommand, DeleteContactCommand, …)
│       │   ├── dispatcher.py (CommandDispatcher)
│       │   └── results.py   (AppResult)
│       ├── storage/         — Збереження на диск
│       │   ├── models.py
│       │   └── pickle_storage.py
│       └── interfaces/      — Способи взаємодії з користувачем
│           ├── cli/         (main.py, parser.py, runner.py, presenter.py)
│           ├── tui/         (main.py через Textual)
│           └── api/         (main.py через FastAPI/Flask)
└── tests/                   — Автотести
```

## Архітектурні шари (Layers)

Потік викликів спрямований зверху вниз:

```text
interfaces → application → services → repositories → domain → storage
```

### 1. Domain Layer (доменні моделі)

- **Що містить:** `Contact`, `Note`, `Phone`, `Email`, `Birthday`, `Tag`.
- **Логіка:** «Ядро» системи. Тут зосереджена основна валідація полів (телефони, пошта, дати). Валідація не розмазується по проєкту, а зосереджена в одному місці.

### 2. Repository Layer (робота з колекціями)

- **Обов’язки:** додавання, пошук, видалення, сортування та отримання об’єктів.
- **Логіка:** працює суто з об’єктами домену і нічого не знає про UI (CLI/TUI/API) чи спосіб збереження на диск. Поточний `internal dict` у майбутньому можна замінити на SQLite чи vector storage.

### 3. Services Layer (бізнес-логіка)

- **Обов’язки:** «Мозок» системи — вирішує, як виконувати команди, які перевірки запускати та які репозиторії викликати.
- **Обмеження:** чиста логіка. Сервіс **не повинен** друкувати текст у консоль, використовувати `input()`, знати про HTTP або фреймворки на кшталт `prompt_toolkit` та `Textual`.

### 4. Storage Layer (збереження даних)

- **Обов’язки:** лише `save` / `load` (взаємодія з диском).
- **Реалізація:** `PickleStorage`, файл за шляхом `~/.personal_assistant/data.pkl`. Ізоляція шару дозволяє перейти на SQLite, додати бекапи або імпорт/експорт.

### 5. Application Layer (система команд)

- **Обов’язки:** єдина точка входу для команд. Інтерфейси (CLI/TUI/API) не звертаються до бізнес-логіки напряму, а викликають однакові команди через загальний шар додатку.

### 6. Interfaces Layer (взаємодія з користувачем)

- **Обов’язки:** вхідні дані (input) та відображення результату (output). Підтримка CLI (з автокомплітом та історією), TUI (`Textual`) та REST API (`FastAPI` / `Flask`).

## Стратегія валідації даних

- **Головне правило:** одне поле = одне місце для нормалізації та валідації. Перевірка відбувається всередині доменних field-класів (`Phone`, `Email` тощо).
- **Навіщо:** уникаємо дублювання перевірок у CLI-парсері, сервісах, репозиторіях чи ендпоїнтах API. Розкидана логіка важко підтримувати.
- **Pydantic:** пакет **не** використовується як головний шар валідації. Його можна додати пізніше як опціональний DTO-шар для зовнішнього формату (схеми API чи форми TUI). Фінальна бізнес-валідація залишається за доменом.

## Готовність до AI (AI-Ready)

Архітектура враховує майбутню інтеграцію штучного інтелекту:

- **Базовий варіант:** простий сервіс підбору тегів за ключовими словами (`KeywordTagSuggestionService`).
- **Перспектива:** embeddings, векторний пошук, локальні LLM, семантичний пошук, ШІ-генерація тегів.
- **Потенційний стек:** SQLite, `sentence-transformers`, `bge-m3`, Ollama, Qwen, ChromaDB.
