# Архітектура Assistant T800

Цей документ описує поточну архітектуру проєкту Assistant T800. Документ відображає фактичну реалізацію поточної версії, а також фіксує напрямок подальшого розвитку.

Головна мета архітектури — зберегти просту навчальну систему, але не змішувати в одному місці введення користувача, бізнес-логіку, збереження даних, локалізацію та роботу зі штучним інтелектом.

---

## 1. Загальна ідея

Assistant T800 — це персональний помічник для роботи з адресною книгою.

Поточна модель проєкту навмисно спрощена: основною сутністю є контакт. Нотатка та теги зараз розглядаються як атрибути контакту, а не як повністю окремі сутності з власними репозиторіями. Це дозволяє закрити вимоги фінального проєкту без надмірного ускладнення структури.

Основний інтерфейс — CLI. Додатково є TUI-режим на Textual із AI-чатом.

---

## 2. Поточні можливості

### Контакти

Система підтримує:

- створення контакту;
- перегляд усіх контактів;
- перегляд одного контакту;
- пошук контактів;
- оновлення адреси;
- оновлення дня народження;
- додавання телефонів;
- додавання e-mail;
- видалення контакту;
- видалення адреси;
- видалення дня народження;
- видалення одного, кількох або всіх телефонів;
- видалення одного, кількох або всіх e-mail.

### Дні народження

Команда `birthdays [days]` показує контакти, яких потрібно привітати протягом заданого періоду. Якщо кількість днів не вказана, використовується 7 днів. Якщо день народження потрапляє на суботу або неділю, дата привітання переноситься на понеділок.

### Пошук

Пошук реалізовано за такими напрямками:

- усі поля контакту;
- ім’я;
- телефон;
- e-mail;
- нотатка;
- тег.

Пошук нечутливий до регістру та працює за частковим збігом.

### Нотатки та теги

У моделі контакту вже є поле нотатки та набір тегів. Пошук за нотатками та тегами вже реалізовано. Команди редагування нотаток і тегів ще плануються.

### Збереження даних

Дані зберігаються у pickle-файлі:

```text
.data/address_book.pkl
```

Історія CLI-команд зберігається окремо:

```text
.data/cli_commands_history
```

## 3. Структура проєкту

```text
src/assistant_t800/
├── ai/                  # AI-інструменти для TUI
├── application/         # система команд, dispatcher, результати, помилки
├── domain/              # доменні моделі та поля
├── interfaces/          # CLI, TUI та launcher
├── localization/        # локалізовані повідомлення
├── repositories/        # колекція контактів і пошук
├── services/            # бізнес-логіка над репозиторієм
├── storage/             # pickle-збереження
├── suggestions/         # fuzzy та AI-підказки команд
├── bootstrap.py         # збірка runtime-об’єктів
└── cli.py               # точка входу пакета
```

## 4. Шари системи

### 4.1. `domain`

Шар домену містить основні моделі та поля:

- `Contact`;
- `Name`;
- `Phone`;
- `Email`;
- `Address`;
- `Birthday`;
- `BirthdaysListContact`.

Саме доменні поля відповідають за фінальну валідацію значень. Наприклад, телефон перевіряється у `Phone`, e-mail — у `Email`, дата народження — у `Birthday`.

Важливий принцип: application layer може визначити, на що схожий аргумент, але остаточна перевірка значення залишається в домені.

### 4.2. `repositories`

Репозиторій `ContactsRepository` зберігає контакти у пам’яті та надає методи для:

- додавання контакту;
- отримання контакту;
- видалення контакту;
- отримання всіх контактів;
- пошуку;
- пошуку найближчих днів народження.

Репозиторій не друкує текст у консоль і не знає про CLI або TUI.

### 4.3. `services`

`ContactsService` є шаром бізнес-операцій над репозиторієм. Він надає методи для команд вищого рівня:

- створення контакту;
- оновлення полів;
- додавання телефонів та e-mail;
- видалення значень;
- очищення полів;
- пошуку;
- пошуку найближчих днів народження.

Сервіс не повинен працювати з `input()` або `print()`.

### 4.4. `application`

Application layer є центром командної системи.

Основні компоненти:

- `Command` — опис команди;
- `CommandDispatcher` — розбір введеного рядка та виклик handler-а;
- `AppContext` — спільний контекст виконання;
- `AppResult` — структурований результат команди;
- `AppError` та `CommandError` — контрольовані помилки.

`CommandDispatcher` підтримує пошук найдовшого збігу команди або alias. Завдяки цьому двослівні aliases, наприклад `знайди телефон`, мають пріоритет над коротким alias `знайди`.

### 4.5. `interfaces`

Шар інтерфейсів відповідає за взаємодію з користувачем.

#### CLI

CLI є основним інтерфейсом. Він відповідає за:

- читання команд;
- історію команд;
- автодоповнення;
- виведення результатів;
- підтвердження небезпечних дій;
- підключення suggestions.

#### TUI

TUI є додатковим інтерфейсом на Textual. Він використовує той самий файл збереження, але має власний AI-чат і власний спосіб відображення.

### 4.6. `localization`

Повідомлення винесені в окремий шар локалізації.

Структура:

```text
localization/
├── messages.py
├── multilang.py
└── locales/
    ├── en.ini
    └── uk.ini
```

Код звертається до локалізованих текстів через `Message.*` або `ErrorCode.*`, а не через рядкові ключі.

### 4.7. `storage`

Збереження реалізовано у `storage/pickle.py` через клас `AssistantStorage`.

Він працює як context manager:

```python
with AssistantStorage(path=pickle_db) as repository:
    ...
```

Під час входу в контекст дані завантажуються, а під час виходу — зберігаються.

### 4.8. `suggestions`

Модуль `suggestions` відповідає за підказки команд.

Він містить:

- переносний registry команд;
- fuzzy-пошук;
- AI fallback;
- resolution service;
- spinner для очікування відповіді AI.

Цей модуль не замінює deterministic command system. Він лише допомагає користувачу перейти від неточного або природного вводу до канонічної команди.

## 5. Потік виконання CLI-команди

```text
Користувач вводить рядок
        ↓
CliRunner
        ↓
CommandDispatcher
        ↓
Application handler
        ↓
ContactsService
        ↓
ContactsRepository
        ↓
Domain model
        ↓
AppResult
        ↓
CliPresenter
        ↓
Вивід у консоль
```

Якщо команда невідома, `CliRunner` може звернутися до `SuggestionService`. Якщо suggestions пропонує команду, вона виконується як звичайний канонічний рядок.

---

## 6. Підтвердження небезпечних дій

Видалення контакту або даних контакту не виконується мовчки.

Приклад:

```text
Видалити контакт «Аліса»? (так/ні):
```

Application handler спочатку повертає `AppResult.confirm(...)`. CLI показує prompt підтвердження. Якщо користувач погодився, команда повторно виконується з прапорцем `confirmed=True`.

---

## 7. Локалізація

Локалізація побудована навколо `MultiLangEnum`.

Приклад:

```python
Message.CONFIRM_REMOVE_CONTACT.render(name="Аліса")
```

або:

```python
ErrorCode.CONTACT_NOT_FOUND.render(name="Аліса")
```

Тексти зберігаються в `.ini` файлах, розділених за класами enum:

```ini
[Message]
WELCOME = Вітаємо в Assistant T800.

[ErrorCode]
CONTACT_NOT_FOUND = Контакт «{name}» не знайдено.
```

## 8. Збереження даних

Поточне збереження просте та достатнє для навчального проєкту:

- формат: pickle;
- файл: `.data/address_book.pkl`;
- об’єкт збереження: `ContactsRepository`;
- автоматичне завантаження при старті;
- автоматичне збереження при завершенні.

У майбутньому pickle можна замінити на SQLite або інший механізм без переписування CLI-команд.

---

## 9. Запланований розвиток

Найближчі логічні кроки:

1. Додати `edit-note <name>`.
2. Додати `remove-note <name>`.
3. Додати `add-tag <name> <tag>`.
4. Додати `remove-tag <name> <tag>`.
5. Додати тести для application layer та CLI-сценаріїв.
6. За потреби розширити TUI так, щоб він також викликав спільний command layer.

---

## 10. Що свідомо не ускладнюємо

Проєкт не намагається бути великою CRM-системою. Ми не додаємо зайві абстракції без практичної потреби.

Свідомо уникаємо:

- окремого репозиторію нотаток на цьому етапі;
- окремого репозиторію тегів на цьому етапі;
- складного механізму міграцій;
- зайвого поділу файлів заради самого поділу;
- перенесення бізнес-логіки в AI.

Головна ідея: deterministic core first, AI as an optional helper.
