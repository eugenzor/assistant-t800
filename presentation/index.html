Потік виконання

04Як рухаються дані

команда→ диспетчер→ use case→ service→ repository→ storage

Жоден шар не лізе через голову сусіда — команди працюють через репозиторій, а не через файл.

Domain-First + AI-fallback

05Валідація живе в домені

  • Валідація в доменних моделях: Phone · Email · Birthday · Address — єдине джерело істини
  • Телефон → канонічний E.164; валідація формату (10–15 цифр)
  • Парсер розпізнає оператора/місто (Kyivstar · Vodafone · Львів…) — ParsedPhone.owner
  • Два набори парсерів: детермінований (services/) + AI-fallback (ai/), оркеструються в application/normalizers.py
  • Детермінований упорався → AI не викликається → миттєво. Інакше AI домислює адресу по частковим даним

Інтерфейс №2 · презентація даних

11Extended CLI (Rich)

  • Вмикається автоматично, коли доступний Rich (інакше — звичайний CLI)
  • Картки контактів, таблиця контактів, таблиця іменинників
  • Кольорові статуси, вітальний екран, інлайн-підказки
  • Адаптивний вивід — звужується на вузьких терміналах
Інтерактивні редактори

12Редагування нотаток і тегів

  • edit-note — багаторядковий редактор нотатки під карткою
  • edit-tags — однорядковий редактор тегів
  • Передзавантаження поточного значення; Ctrl+S — зберегти, Esc — скасувати
AI-режим · два підходи

13Асистент «Арні»

  • CLI → prompt-модель: один точний запит → швидко й однозначно
  • TUI → агент «Арні»: сам обирає інструменти → думає довше, без 100% гарантії
  • Модель Gemini 3.1 Flash Lite через pydantic-ai; інструменти = методи ContactsService 1:1
  • AI не пише в сховище напряму — лише через сервіс
AI + приватність

14suggest-tags

  • suggest-tags — AI пропонує до 7 тегів, користувач підтверджує
  • Маскує телефон (+XXX*****), не шле рядок адреси в модель
  • Без підтвердження нічого не зберігається
